# Repository Audit — Reference Prompt

Вставить этот файл целиком в начало аудита любому AI.  
Перед запуском — адаптировать блок «Контекст проекта» под реальный стек.

---

## Перед началом — синхронизация

Аудит на устаревшем snapshot бесполезен. До чтения кода:
1. Прочитать последние 10 коммитов: `git log --oneline -10 main`
2. Зафиксировать HEAD: `git rev-parse main` — указать SHA в начале отчёта
3. Пробежать по `tasks/lessons.md` и `git log --grep=fix` за последний месяц — не повторять уже починенное

Если найден баг — убедиться что он **есть в текущем HEAD**, а не в кеше.

## Контекст проекта ← заполнить перед аудитом

```
Тип проекта: [бот / API / CLI / веб-сервис / библиотека / ...]
Стек: [...]
Главный язык: [...]
Внешние API: [...]
CI/CD: automerge.yml (claude/** → main)
SSOТ этого проекта: [файл или модуль — единственный источник правды]
Вторичные источники: [что должно следовать из SSOT]
```

**НЕ проверять** (нерелевантно для personal tool):
- multi-user изоляция и RBAC
- rate limiting и abuse prevention
- GDPR / compliance
- Docker / Kubernetes / horizontal scaling

## Pipeline — 3 pass

Один pass не справляется с объёмом. Запускать как отдельные сессии:

| Pass | Секции | Фокус |
|------|--------|-------|
| 1 — Корректность | §1, §2, §3, §8 | SSOT sync, слои, внешние API, обработка ошибок |
| 2 — Безопасность + документация | §6, §7, §5, §5.1, §5.2 | security, dead code, docs vs reality |
| 3 — Архитектура + CI | §4, §9, §10, §11 | CI/CD, наблюдаемость, зависимости, дизайн |

Каждый pass — свой мини-отчёт в формате §«Формат отчёта».

## Чеклист аудита

### 1. СИНХРОНИЗАЦИЯ (SSOT → вторичные источники)

Определить SSOT проекта (из блока «Контекст») и проверить что все вторичные источники ему соответствуют:

- [ ] Список сущностей (команды / роуты / модели / режимы) одинаков во всех местах где упомянут
- [ ] Нет места где список задан вручную и может разойтись с SSOT
- [ ] Добавление новой сущности требует правки только SSOT — всё остальное следует автоматически или через CI-гейт
- [ ] `scripts/check_consistency.py` проверяет ключевые расхождения между SSOT и вторичными источниками

### 2. ВНЕШНИЕ API И КЛИЕНТЫ

Для каждого внешнего API / сервиса:
- [ ] ошибка → исключение, НЕ `return None` / silent fail
- [ ] таймаут задан явно
- [ ] retry есть там где нужен (сетевые ошибки, 429, 5xx)
- [ ] API-ключи только из env, не из кода
- [ ] версия API задокументирована — есть риск deprecation?

### 3. ЧИСТОТА СЛОЁВ

- [ ] Каждый слой имеет одну ответственность — назвать её одним предложением
- [ ] Нет доменной логики в точке входа (main.py / index.ts / app.py)
- [ ] Нет захардкоженных путей / URL / строк вне конфига
- [ ] Добавление новой фичи требует правки скольких файлов? >5 = слишком связно

### 4. CI/CD

- [ ] Тесты запускаются перед merge в main
- [ ] Merge не происходит при падении тестов
- [ ] `automerge.yml` триггер ограничен только нужными ветками (`claude/**`, `cursor/**`) — не срабатывает на всё подряд
- [ ] `scripts/check_consistency.py` запускается в CI
- [ ] Нет `--no-verify`, нет force push в main

### 5. ДОКУМЕНТАЦИЯ vs РЕАЛЬНОСТЬ

**Сканировать ВСЕ `.md` файлы в репо:**
```
find . -name '*.md' -not -path './.git/*' -not -path '*/archive/*'
```

- [ ] `README.md` описывает реальную структуру, не устаревшую
- [ ] `NEW_PROJECT.md` заполнен — нет незакрытых плейсхолдеров `[...]`
- [ ] `tasks/lessons.md` актуален — нет паттернов без даты или без правила
- [ ] Нет упоминаний удалённых модулей / файлов / веток
- [ ] Git workflow описан одинаково везде где упомянут (адаптер, SYSTEM.md, README)
- [ ] Все пути упомянутые в .md реально существуют в репо

#### 5.1 ПЕРЕКРЁСТНАЯ СОГЛАСОВАННОСТЬ АДАПТЕРОВ

Для каждого AI-инструмента который использует этот проект:
- [ ] Есть файл адаптера (`CLAUDE.md` / `.cursorrules` / аналог)
- [ ] Git workflow описан одинаково в адаптере и в `automerge.yml`
- [ ] Адаптер описывает только СВОЙ инструмент — не чужой
- [ ] `SYSTEM.md` загружается корректно для каждого инструмента

#### 5.2 ГЛОБАЛЬНАЯ ПЕРЕКРЁСТНАЯ СВЕРКА

| Что сверять | Откуда | Куда |
|---|---|---|
| Список ключевых сущностей | SSOT (код) | README, docs, комментарии |
| Git workflow | `automerge.yml` | адаптер, `SYSTEM.md`, `QUICKSTART.md` |
| Внешние зависимости | `requirements.txt` / `package.json` | `README.md`, `NEW_PROJECT.md` |
| Структура папок | реальный репо | `README.md`, `NEW_PROJECT.md` |

- [ ] Число ключевых сущностей одинаково во всех источниках
- [ ] Все пути в .md совпадают с реальной структурой
- [ ] Git workflow согласован во всех файлах где упомянут

### 6. БЕЗОПАСНОСТЬ (минимум для personal tool)

- [ ] API ключи только в `.env`, нигде в коде / тестах / логах
- [ ] `.env` в `.gitignore`, нет `.env` в истории git
- [ ] Logger не печатает значения env переменных и `Authorization` заголовков
- [ ] Path traversal: пользовательские пути проходят валидацию перед использованием
- [ ] `subprocess.run()` без `shell=True`, аргументы как список
- [ ] Пользовательский ввод не попадает в `exec()` / `eval()` / SQL без sanitization

### 7. МЁРТВЫЙ КОД

- [ ] Все зависимости в `requirements.txt` / `package.json` реально используются
- [ ] Нет workflow файлов с `branches: [__never__]` или аналогом
- [ ] `README.md` не ссылается на удалённые модули
- [ ] Нет закомментированного кода старше одного коммита без объяснения

### 8. ОБРАБОТКА ОШИБОК (fail-loud)

- [ ] Нет `try: ... except Exception: pass` в основном коде
- [ ] Ошибки идут в stderr и/или логгер — не теряются
- [ ] Некорректный пользовательский ввод даёт понятное сообщение, не traceback
- [ ] Отсутствие обязательного env-переменного обнаруживается при старте, не в рантайме

### 9. НАБЛЮДАЕМОСТЬ

- [ ] Есть логирование (в файл и/или консоль)
- [ ] Каждый запуск идентифицируем (timestamp или ID)
- [ ] Ошибки не теряются — идут через `logger.error`, не через голый `print()`
- [ ] Нет log spam — один запрос = разумное количество строк

### 10. ЗАВИСИМОСТИ

- [ ] Нет известных CVE в продовых пакетах (`pip-audit` / `npm audit`)
- [ ] Версии имеют ограничения (`<2.0.0`), не просто `>=1.0`
- [ ] CI ставит зависимости из lock-файла, не хардкодит пакеты в workflow

### 11. АРХИТЕКТУРНЫЙ СМЫСЛ

Не про корректность — про дизайн.

- [ ] Что можно удалить без потери функциональности — назвать конкретно или сказать «нет таких»
- [ ] Что сделано «на вырост» и не используется — конкретные функции/классы
- [ ] Где есть дублирование логики — обосновано или копипаста
- [ ] Добавление новой фичи — сколько файлов трогать? Назвать их

**Формат для §11 отличается:** не «файл:строка → фикс», а «паттерн X встречается в N местах — обосновано / упростить так-то».

### 12. AUDIT FRESHNESS

После завершения:
- [ ] Указать HEAD main SHA в начале отчёта
- [ ] Если пункт чеклиста «уже починено» — отметить, не выписывать как новую проблему

## Формат отчёта

**Калибровка severity до написания:**

```
SEVERITY:
  BLOCKER  = потеря данных / runtime не работает / дыра в безопасности
  HIGH     = silent degradation / неверный результат / неверный биллинг
  MEDIUM   = риск maintainability / drift который выстрелит через месяц
  LOW      = косметика / расхождение в docs / стиль

CONFIDENCE:
  HIGH    = нашёл в коде, строку указал, воспроизводимо
  MEDIUM  = паттерн виден, точная строка не проверена
  LOW     = подозрение — помечать явно, не выписывать как факт
```

Каждая проблема строго в формате:

```
[SEVERITY] [CONFIDENCE]
Файл: path/to/file.py:line
Проблема: что конкретно не так
Последствие: что сломается в реальном использовании
Фикс: конкретное исправление (не "рассмотреть", а что именно сделать)
```

Завершить отчёт тремя блоками:
1. **Блокеры** — что мешает работе прямо сейчас
2. **Что сделано хорошо** — не пропускать
3. **Следующие 3 приоритета** — конкретные задачи в порядке важности

**Не писать:**
- общие советы без привязки к файлу и строке
- «рассмотреть использование паттерна X»
- enterprise-рекомендации (RBAC, scaling и т.д.)
